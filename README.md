## Memora – система интервального повторения

Memora – это веб‑приложение для запоминания информации с помощью карточек и алгоритма интервальных повторений SM‑2. Проект состоит из backend‑части на Django REST Framework и frontend‑части на React.

---

## Общая архитектура

- **Backend**: Django + Django REST Framework  
  - Отвечает за авторизацию, API для колод/карточек, обучение, расписание и статистику.
- **Frontend**: React (SPA)  
  - Отвечает за интерфейс: страницы входа/регистрации, колоды, обучение, дашборд, статистику, расписание и FAQ.

Back и front общаются по HTTP через JWT‑авторизацию (`access_token` / `refresh_token` в `localStorage` на фронте).

## Структура backend

Корневая папка backend:  
`back/`

### Приложение `config` (основные настройки проекта)

- `back/config/settings.py`
  - Базовые настройки Django.
  - Подключение приложений: `accounts`, `cards`, `pet`.
  - Настройки DRF, JWT, CORS и т.п.
  - Подключение кастомного обработчика ошибок:
    - `REST_FRAMEWORK['EXCEPTION_HANDLER'] = 'config.exceptions.custom_exception_handler'`.

- `back/config/urls.py`
  - Основные маршруты API:
    - `path('api/', include('accounts.urls'))`
    - `path('api/', include('cards.urls'))`
    - `path('api/', include('pet.urls'))`
  - Эндпоинты JWT (получение/обновление токенов).

- `back/config/exceptions.py`
  - `custom_exception_handler` – единый формат ошибок для DRF.

- `back/config/asgi.py`, `back/config/wsgi.py`
  - Стандартные файлы запуска ASGI/WSGI.

### Приложение `accounts` (пользователи и профиль)

Папка: `back/accounts/`

- `models.py`
  - `User` – кастомная модель пользователя (заменяет стандартного `auth.User`).
  - `UserProfile` – профиль пользователя:
    - Статистика: общее количество выученных карточек, очки, streak (текущая и максимальная серия), последняя дата занятий и уровень.
  - Сигналы для автоматического создания/сохранения `UserProfile` при создании пользователя.

- `serializers.py`
  - `UserSerializer` – сериализация пользователя.
  - `UserRegistrationSerializer` – регистрация нового пользователя.
  - Другие вспомогательные сериализаторы для профиля (если есть).

- `views.py`
  - ViewSet’ы/представления для:
    - Регистрации пользователя.
    - Получения/обновления текущего пользователя (`/api/accounts/me/`).

- `urls.py`
  - Маршрутизация для `accounts` (регистрация, профиль и т.п.).

### Приложение `cards` (колоды, карточки, обучение, статистика)

Папка: `back/cards/`

- `models.py`
  - `Folder` – папка для организации колод:
    - Иерархия папок (родитель/дочерние), цвет, иконка, описание.
  - `Deck` – колода:
    - Пользователь, папка, название, описание, цвет, флаг публичности.
  - `Card` – карточка:
    - Колода, текст вопроса (`front`), текст ответа (`back`).
    - Поля для алгоритма SM‑2: `ease_factor`, `interval`, `repetitions`, `next_review`, `last_reviewed`, `is_suspended`.
  - `StudySession` – сессия обучения:
    - Пользователь, колода (опционально), режим (обычный / практика), флаг реверса, статистика по сессии.
  - `CardReview` – отдельный отзыв/оценка карточки в рамках сессии:
    - Карточка, сессия, оценка, время ответа, изменения факторов SM‑2.

- `serializers.py`
  - `CardSerializer`:
    - Упрощённая версия: только вопрос/ответ и служебные поля (без тегов и изображений).
  - `DeckSerializer`:
    - Краткие данные по колоде + количество карточек (`cards_count`) и имя папки.
  - `FolderSerializer`, `FolderTreeSerializer`:
    - Для дерева папок и отображения их содержимого.
  - `CardReviewSerializer`, `StudySessionSerializer`:
    - Сериализация отзывов и сессий обучения.

- `views.py`
  - `DeckViewSet` (`/api/decks/`):
    - CRUD по колодам текущего пользователя.
    - Фильтрация, сортировка, поиск.
    - Экшен `cards` для получения карточек конкретной колоды (`/api/decks/{id}/cards/`).
  - `CardViewSet` (`/api/cards/`):
    - CRUD по карточкам пользователя.
    - Фильтрация по колоде, статусу (новые/изучаемые/освоенные), тегам (если останутся), текстовый поиск.
    - `perform_create` привязывает колоду только из колод текущего пользователя.
    - Экшен `popular_tags` – если в будущем снова будут теги.
  - `StudyViewSet` (`/api/study/...`):
    - `due_cards` – карточки, срок повторения которых уже наступил.
    - `all_cards` – случайный набор карточек колоды для практики.
    - `matching_cards` – карточки для режима "Подбор пар".
    - `test_cards` – карточки для режима "Тест" с вариантами ответов.
    - `start_session` – создать сессию обучения.
    - `submit_review` – сохранить оценку карточки и применить SM‑2.
    - `end_session` – завершить сессию, начислить опыт питомцу.
    - `schedule` – расписание повторений по дням (для календаря/расписания).
    - `stats` – базовая статистика (количество карточек и т.п.).
  - `StatisticsViewSet` (`/api/statistics/...`):
    - `dashboard` – агрегированная статистика для дашборда.
    - `learning_stats` – активность по дням.
    - `tags_stats` – статистика по тегам (для будущего расширения).
    - `decks_progress` – прогресс по колодам.
  - `FolderViewSet` (`/api/folders/...`):
    - CRUD по папкам, древовидное представление, перемещение папок.

- `urls.py`
  - DRF‑роутеры для `DeckViewSet`, `CardViewSet`, `StudyViewSet`, `StatisticsViewSet`, `FolderViewSet`.

- `sm2.py`
  - Реализация алгоритма SM‑2: расчёт интервалов, коэффициента лёгкости и т.п.

### Приложение `pet` (учебный питомец)

Папка: `back/pet/`

- `models.py`
  - `StudyPet` – виртуальный питомец:
    - Пользователь, тип питомца, уровень, опыт, streak и т.д.
    - Методы для добавления XP и обновления streak.

- `serializers.py`
  - `StudyPetSerializer` – сериализация состояния питомца.

- `views.py`
  - `PetViewSet` (`/api/pet/`):
    - Получение текущего питомца.
    - Обновление параметров (`/api/pet/update_pet/`).
    - Начисление XP (`/api/pet/xp/`), вызывается при завершении сессий обучения.

- `urls.py`
  - Маршрутизация для `PetViewSet`.

## Структура frontend

Корневая папка фронтенда:  
`front/`  
Основной код: `front/src/`

### Входная точка и маршрутизация

- `front/src/main.jsx`
  - Подключение React, корневой компонент `App`.

- `front/src/App.jsx`
  - Настройка роутинга (React Router).
  - Публичные маршруты:
    - `/` – главная (лендинг).
    - `/faq` – часто задаваемые вопросы.
    - `/login` – вход.
    - `/register` – регистрация.
  - Защищённые маршруты (требуют JWT‑авторизации):
    - `/dashboard` – дашборд.
    - `/decks` – список колод.
    - `/decks/:id` – детальная страница колоды и режимы обучения.
    - `/study` – разные режимы обучения (по query‑параметрам).
    - `/statistics` – аналитика и статистика.
    - `/schedule` – календарь и расписание повторений.

### Сервис доступа к API

- `front/src/services/api.js`
  - `api` – основной экземпляр `axios` (JSON):
    - Базовый URL: `http://localhost:8000/api`.
    - В каждый запрос подставляет `Authorization: Bearer <access_token>`, если токен есть в `localStorage`.
    - Общий интерцептор ответов:
      - При `401` пытается обновить токен через `/token/refresh/`.
      - В случае неудачи очищает токены и перенаправляет на `/login`.
  - `apiFormData` – экземпляр `axios` для `multipart/form-data` (если понадобится).
  - `cardsAPI`:
    - `getCards`, `getCard`, `createCard`, `updateCard`, `deleteCard`, `getPopularTags`.
  - `petAPI`:
    - `get`, `update`, `addXP`.

## Основные пользовательские сценарии

1. **Регистрация и вход**
   - Пользователь создаёт аккаунт или входит.
   - Фронтенд получает `access_token` и `refresh_token` и сохраняет их в `localStorage`.
   - Все защищённые страницы становятся доступными.

2. **Создание колоды и карточек**
   - На странице `Колоды` создаётся новая колода.
   - На странице колоды добавляются карточки (только вопрос и ответ).

3. **Обучение**
   - На странице колоды выбирается режим:
     - Обучение – интервальное повторение с оценками и обновлением SM‑2.
     - Тренировка – простой просмотр карточек.
     - Подбор пар – игровой формат.
     - Тест – вопросы с вариантами ответов.
   - Backend создаёт сессию (при необходимости), отдаёт карточки и принимает оценки.

4. **Дашборд и статистика**
   - Дашборд показывает сводные цифры и ближайшие задачи.
   - Страница статистики даёт детальный срез: сколько учились, какие колоды активнее и т.п.

5. **Расписание и календарь**
   - Раздел "Расписание" и календарь на дашборде помогают увидеть нагрузку вперёд:
     - Сколько карточек сегодня, завтра и т.д.
     - Распределение по колодам.

6. **Учебный питомец**
   - Питомец растёт по мере выполнения сессий обучения.
   - Backend начисляет опыт при завершении сессий, фронтенд отображает текущий уровень и XP.

## Локальный запуск (общая идея)

Точный список команд зависит от того, как настроена среда, но в целом:

1. **Backend**
   - Установить зависимости (например, через `pip install -r requirements.txt`).
   - Применить миграции: `python manage.py migrate`.
   - Запустить сервер: `python manage.py runserver`.

2. **Frontend**
   - Перейти в папку `front/`.
   - Установить зависимости: `npm install`.
   - Запустить dev‑сервер: `npm run dev`.

Адрес фронтенда по умолчанию – что‑то вроде `http://localhost:5173`, backend – `http://localhost:8000/api`.

## Как ориентироваться в коде

- За бизнес‑логику и структуру данных отвечают файлы в `back/cards/` и `back/accounts/`.
- За алгоритм SM‑2 – `back/cards/sm2.py` и вызовы в `StudyViewSet.submit_review`.
- За все REST‑эндпоинты – ViewSet’ы и роутеры в `back/cards/views.py`, `back/cards/urls.py`, `back/accounts/views.py`, `back/pet/views.py`.
- На фронтенде:
  - Страницы лежат в `front/src/pages`.
  - Повторно используемые компоненты – в `front/src/components`.
  - Все HTTP‑вызовы централизованы в `front/src/services/api.js`.

Если нужно добавить новый функционал, обычно:
- Добавляем модель/поле и миграцию на бэке.
- Обновляем сериализатор и ViewSet.
- Добавляем функции в `api.js` (или используем уже существующие `api`/`cardsAPI`/`petAPI`).
- Добавляем/изменяем страницу или компонент на фронтенде.
